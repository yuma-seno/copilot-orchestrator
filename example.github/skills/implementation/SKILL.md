# Implementation

コードを書く前に既存のパターンを理解し、既存のスタイルに合わせて実装する。

## 実装前チェック

1. **既存コードの調査** — 同等の機能・パターンがないか確認する。ある場合はそれに合わせる
2. **テストの確認** — 対象コードの既存テストを確認。テストがある場合はまず通ることを確認
3. **規約の把握** — リンター設定、フォーマッター設定、既存の命名パターンを確認

## コーディングチェックリスト

| チェック | 基準 |
|---------|------|
| 既存スタイル準拠 | プロジェクトの命名規約・構造パターンに合致しているか |
| 単一責任 | 1つの関数は1つのことだけ行っているか |
| 早期リターン | ガード節でネストを浅くしているか |
| 不変性 | `const` / `readonly` / 不変データ構造を優先しているか |
| エラーハンドリング | エラーを握りつぶしていないか。適切にキャッチまたは伝播しているか |

## セキュリティチェック

実装中に必ず確認する：

- [ ] ユーザ入力のバリデーション・サニタイズ
- [ ] SQLインジェクション、XSS、CSRF対策
- [ ] 機密情報（パスワード、APIキー）のハードコード禁止
- [ ] パストラバーサル対策
- [ ] 適切な権限チェック

## 実装後チェック

- [ ] ビルドが通る
- [ ] リンターが通る
- [ ] 既存テストが全て通る
- [ ] デバッグコードが残っていない（`console.log`, `print`, `debugger`, `TODO`）
- [ ] 使わないコード・インポートがない

## Red Flags — 今すぐ立ち止まる

| 兆候 | 対処 |
|------|------|
| 「このプロジェクトではこうすべき」と思った | 既存コードで実際にどうしているか確認する |
| 「念のため抽象化しておこう」 | YAGNI。現在の要件に必要な最小限に留める |
| 「テストが壊れたのでテストを修正」 | テストではなく実装を見直す |
| 「コメントで説明すれば大丈夫」 | コードを明確にして、コメントを不要にする |
