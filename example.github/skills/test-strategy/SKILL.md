# Test Strategy

テスト計画の立案からテストコードの作成、実行、結果分析まで行う。

**コア原則:** 実装コードを変更する前にテストを書く。テストが失敗する理由を確認してから実装する。

## When to Use

- 新機能・バグ修正のテストを書くとき
- テスト計画を立案するとき
- テストカバレッジを改善するとき

**使わない場面:** コード実装 → code-implementation / コードレビュー → code-review

## チェックリスト

1. **テスト対象分析** — 公開インターフェース、入出力パターン、副作用、依存関係、エラーケース
2. **テスト種別選択** — 適切なテストレベルを選択
3. **テストケース設計** — 正常系・異常系・境界値を設計
4. **テストコード作成** — 実装前にテストを書く（TDD）
5. **実行と検証** — 全テスト通過、カバレッジ確認

## テスト種別

| 種別 | 対象 | 目的 | 速度 |
|------|------|------|------|
| 単体テスト | 関数・メソッド | ロジック正確性 | 高速 |
| 結合テスト | モジュール間連携 | コンポーネント連携 | 中速 |
| E2Eテスト | システム全体 | ユーザ視点の動作 | 低速 |

**優先順位:** 単体テスト → 結合テスト → E2E

## テストケース設計

### 正常系

基本動作、複数の有効入力パターン、オプション引数の組み合わせ

### 異常系

| パターン | 例 |
|---------|-----|
| 無効な入力 | null, undefined, 空文字列, 型違い |
| 境界値 | 最小値, 最大値, 0, 負の数 |
| エラー状態 | ネットワークエラー, タイムアウト, 権限不足 |
| 並行性 | 同時実行, レースコンディション |

## Good Tests vs Bad Tests

<Good>
```typescript
test('空メールでフォーム送信するとエラーが返る', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```
明確な名前、1つの振る舞い、実際のコードをテスト
</Good>

<Bad>
```typescript
test('test1', async () => {
  const mock = jest.fn().mockResolvedValue('ok');
  await submitForm(mock);
  expect(mock).toHaveBeenCalled();
});
```
曖昧な名前、モックの動作をテスト、実際の振る舞いを検証していない
</Bad>

## Red Flags

| 思考 | 現実 |
|------|------|
| 「シンプルすぎてテスト不要」 | シンプルなコードも壊れる。テストは30秒で書ける |
| 「あとでテストを書く」 | あとで書いたテストは即座にパスし、何も証明しない |
| 「手動でテスト済み」 | 手動テストは再現不能。変更のたびに再テスト必要 |
| 「モックだらけで仕方ない」 | 設計が結合しすぎ。DI（依存注入）を検討する |

## Common Mistakes

| ミス | 対策 |
|------|------|
| 実装後にテストを書く | テストが即パスし、検証能力が不明。先に書く |
| テスト名が曖昧 | 振る舞いを説明する名前にする |
| モックの動作をテスト | 実際のコードの振る舞いをテストする |
| 1テストで複数の振る舞い | 1テスト1アサーション（概念的に） |
