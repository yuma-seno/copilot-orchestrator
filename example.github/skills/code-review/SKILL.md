# Code Review

コード品質、ロジック正確性、セキュリティ、パフォーマンス、保守性を評価し、具体的な指摘と改善案を示す。

**コア原則:** 客観的に、具体的に、建設的に。問題点だけでなく良い点にも言及する。

## When to Use

- コードを品質評価するとき
- PRをレビューするとき
- 実装完了後の品質チェック

**使わない場面:** コード実装 → code-implementation / テスト作成 → test-strategy

## レビュー観点

### 1. コード品質

| チェック | ポイント |
|---------|---------|
| 命名 | 目的が読み取れるか。規約に準拠しているか |
| 関数の長さ | 長すぎないか。分割すべきか |
| 単一責任 | 1つの関数/クラスが1つの責務か |
| 重複コード | コピペコードがないか |
| 不要コード | 未使用の変数・インポート・コメントアウト |

### 2. ロジック正確性

| チェック | ポイント |
|---------|---------|
| 要件充足 | 変更が要件を正しく満たしているか |
| エッジケース | null, 空文字列, 空配列, 境界値の処理 |
| 非同期処理 | async/await, Promise のエラーハンドリング |
| レースコンディション | 並行性の問題がないか |

### 3. セキュリティ（重要度: 高）

入力バリデーション / SQLインジェクション / XSS / CSRF / 認証・認可 / 機密情報のハードコード / パストラバーサル / ログへの機密情報出力

### 4. パフォーマンス

O(n²) 以上の不必要な計算量 / N+1問題 / 不必要なメモリ保持 / 並列実行可能な処理の直列実行

### 5. 保守性

可読性 / テスタビリティ / 拡張性 / エラーメッセージの有用性

## 指摘の分類

| 分類 | 基準 | 対応 |
|------|------|------|
| **CRITICAL** | セキュリティ脆弱性、データ損失、バグ | 修正必須。マージ前に対応 |
| **WARNING** | 品質問題、パフォーマンス懸念、テスト不足 | 修正推奨 |
| **INFO** | スタイル改善、代替案、ドキュメント補足 | 検討推奨。対応任意 |

## レビューレポートフォーマット

```markdown
# レビューレポート

## サマリ
- レビュー対象: [ファイルリスト]
- 総合評価: [APPROVE / REQUEST_CHANGES / COMMENT]
- 指摘数: CRITICAL: [数] / WARNING: [数] / INFO: [数]

## CRITICAL（修正必須）
#### C-001: [タイトル]
- ファイル: [パス:行番号] / 問題: [...] / 理由: [...] / 修正案: [コード例]

## WARNING（修正推奨）
#### W-001: [タイトル]
- ファイル: [パス:行番号] / 問題: [...] / 修正案: [...]

## INFO（検討推奨）
#### I-001: [タイトル]
- ファイル: [パス:行番号] / 提案: [...]

## 良い点
- [具体的な良い設計・実装について]
```

## Common Mistakes

| ミス | 対策 |
|------|------|
| 個人の好みで指摘する | 品質・安全性・保守性の観点で判断する |
| 問題の指摘だけ | 修正案をコード例付きで示す |
| 良い点に言及しない | 良い実装・設計にも具体的に触れる |
