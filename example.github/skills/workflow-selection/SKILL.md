# ワークフロー選定スキル

指示内容を分析し、最適な開発ワークフローを選定します。

## 指示タイプの判別

ユーザの指示を以下のカテゴリに分類してください：

### カテゴリ一覧

| カテゴリ | 判別キーワード・特徴 | 例 |
|---------|--------------------|----|
| **新規開発** | 「作成」「追加」「新しい」「実装」「機能追加」 | 「ユーザ認証機能を作成して」 |
| **バグ修正** | 「修正」「バグ」「エラー」「動かない」「おかしい」 | 「ログイン時にエラーが出る」 |
| **リファクタリング** | 「リファクタ」「整理」「改善」「最適化」「分割」 | 「このクラスを分割して」 |
| **調査・分析** | 「調査」「調べて」「なぜ」「原因」「影響範囲」 | 「パフォーマンスが遅い原因を調べて」 |
| **ドキュメント** | 「ドキュメント」「README」「説明」「手順書」 | 「APIのドキュメントを書いて」 |
| **テスト** | 「テスト」「カバレッジ」「検証」 | 「テストを追加して」 |
| **設定・環境** | 「設定」「環境」「CI」「デプロイ」「インフラ」 | 「CI/CDパイプラインを設定して」 |
| **レビュー** | 「レビュー」「確認」「チェック」 | 「このPRをレビューして」 |

### 複合指示の判別

ひとつの指示に複数のカテゴリが含まれる場合：

1. 主目的のカテゴリを特定する（最も重要な成果物を生むカテゴリ）
2. 副次的なカテゴリをサブタスクとして整理する
3. 依存関係を考慮してフェーズの順序を決定する

## ワークフローパターン

### パターン1: フルサイクルフロー（新規開発）

大規模な新規機能の開発に使用します。

```
調査 → 要件定義 → 設計 → 実装 → テスト → レビュー → 完了
```

**使用条件：**
- 新しい機能やコンポーネントの作成
- 複数ファイルにまたがる変更
- 要件の明確化が必要

**各フェーズの目的：**
1. **調査**: 既存コードベースの構造把握、技術スタックの確認、影響範囲の特定
2. **要件定義**: 機能要件・非機能要件の整理、受け入れ基準の定義
3. **設計**: アーキテクチャ決定、コンポーネント設計、データフロー設計
4. **実装**: コードの実装、基本動作確認
5. **テスト**: テストコード作成、テスト実行、カバレッジ確認
6. **レビュー**: コード品質チェック、セキュリティチェック、設計適合性確認

### パターン2: 修正フロー（バグ修正）

既知の問題の修正に使用します。

```
調査 → 原因特定 → 修正実装 → テスト → レビュー → 完了
```

**使用条件：**
- エラーやバグの報告
- 既存機能の不具合修正
- パフォーマンス問題の解消

**各フェーズの目的：**
1. **調査**: 問題の再現確認、関連コードの特定
2. **原因特定**: 根本原因の分析、影響範囲の確認
3. **修正実装**: 修正コードの実装、リグレッション確認
4. **テスト**: 修正の検証テスト、既存テストの確認
5. **レビュー**: 修正の妥当性確認

### パターン3: 改善フロー（リファクタリング）

コード品質の改善に使用します。

```
分析 → 設計 → 段階的改善 → テスト → レビュー → 完了
```

**使用条件：**
- コードの可読性・保守性の改善
- 構造の最適化
- 技術的負債の解消

**各フェーズの目的：**
1. **分析**: 現状のコード構造分析、改善ポイントの特定
2. **設計**: 改善後の構造設計、移行計画の作成
3. **段階的改善**: 小さな単位での変更実施（動作を壊さない）
4. **テスト**: 既存テストの全通過確認、必要に応じてテスト追加
5. **レビュー**: 改善の妥当性確認

### パターン4: 調査フロー（調査・分析）

情報収集と分析が主目的の場合に使用します。

```
情報収集 → 分析 → レポート作成 → 完了
```

**使用条件：**
- 技術調査や影響範囲分析
- パフォーマンス分析
- 技術選定

### パターン5: ドキュメントフロー（ドキュメント作成）

ドキュメントの作成・更新が主目的の場合に使用します。

```
対象理解 → 構成設計 → 執筆 → レビュー → 完了
```

**使用条件：**
- README、APIドキュメント、手順書の作成
- 既存ドキュメントの更新

### パターン6: クイック修正フロー

小規模で明確な変更に使用します。

```
実装 → テスト → 完了
```

**使用条件：**
- 変更内容が明確で限定的
- 単一ファイルの修正
- 要件に曖昧さがない

## 選定の判断フロー

```
指示を受信
  ↓
指示タイプを判別
  ↓
不確実性の評価
  ├─ 高い（要件が不明確、技術的に未知） → フルサイクルフロー or 調査フロー
  ├─ 中程度（概ね明確だが一部不明） → 修正フロー or 改善フロー
  └─ 低い（完全に明確） → クイック修正フロー
  ↓
規模の評価
  ├─ 大規模（複数コンポーネント） → フルサイクルフロー
  ├─ 中規模（複数ファイル） → 該当カテゴリのフロー
  └─ 小規模（単一ファイル） → クイック修正フロー or 該当カテゴリのフロー
  ↓
フローを確定し、todoツールでタスクリストを作成
```

## 注意事項

- 判断に迷った場合は、ユーザに `askQuestion` で確認する
- 作業途中でフローの変更が必要になった場合は、柔軟に調整する
- 各フェーズの完了条件は `skills/quality-gate/SKILL.md` を参照する
