# Design

要件に基づいて実装可能な設計を行う。既存コードベースのパターンを最優先で尊重する。

<HARD-GATE>
設計を始める前に、既存コードベースで使われているパターン・構造・命名規約を調査する。未調査のまま設計してはならない。
</HARD-GATE>

## 設計プロセス

### 1. 既存パターンの調査

- 同等の機能はどう実装されているか
- モジュール分割、レイヤー構成、依存の方向
- 命名規約、ファイル配置規約
- エラーハンドリングのパターン

### 2. コンポーネント設計

各コンポーネントに以下を定義する：

```markdown
### [コンポーネント名]
- 責務: [単一責任に基づく責務]
- 入力: [型と説明]
- 出力: [型と説明]
- 依存: [依存先と理由]
- エラー: [エラー条件 → ハンドリング方針]
```

### 3. 設計判断の記録

判断が必要な箇所は、選択肢と選択理由を記録する：

| 判断事項 | 選択肢 | 選択 | 理由 |
|---------|--------|------|------|
| [何を決めたか] | A: [...], B: [...] | [A or B] | [なぜその選択か] |

## 設計原則

| 原則 | 行動 |
|------|------|
| 既存パターンの尊重 | プロジェクトで使われているパターンを調査し優先する |
| YAGNI | 現在の要件に必要な最小限の設計に留める |
| 単一責任 | 各コンポーネントは1つの責務のみ |
| テスタビリティ | 依存の注入・純粋関数を活用し、テストしやすくする |
| 依存は内向き | 外部(UI/DB)から内部(ビジネスロジック)に依存させる |

## Red Flags

| 兆候 | 対処 |
|------|------|
| 「将来の拡張に備えて抽象化」 | 今必要ないなら作らない |
| 「新しいパターンの方が優れている」 | 既存パターンと混在させない |
| 「設計パターンを適用しよう」 | 問題がないのにパターンを適用しない |
| 「テストは後で考える」 | テスト困難な設計は今修正する |

## 成果物

1. 概要
2. 既存パターンの調査結果
3. コンポーネント設計
4. データフロー（正常系・異常系）
5. インターフェース定義
6. 設計判断の記録
7. 影響範囲（新規作成ファイル、変更ファイル）
