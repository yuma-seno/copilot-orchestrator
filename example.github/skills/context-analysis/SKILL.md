# Context Analysis

タスク完了後に作業内容を振り返り、コードベースと作業プロセスの具体的な改善提案を導出する。

**コア原則:** 事実に基づいて分析する。一般論ではなく、このタスクで実際に観察した問題から改善案を導く。

## When to Use

- タスク完了後の振り返り（サブエージェントの作業フローの最終ステップ）
- セッション全体の振り返り
- コードベースの改善機会の特定

## 分析プロセス

### 1. 事実の収集

以下を確認し、事実を列挙する：

- **変更内容** — 何を・どのファイルに・どう変更したか
- **遭遇した問題** — 予想外の動作、理解に時間がかかった箇所、回り道した箇所
- **判断した箇所** — 複数の選択肢があった場面と選択理由
- **既存コードの状態** — 触れたコードの品質、パターン、技術的負債

### 2. パターンの特定

収集した事実から以下のパターンを探す：

| 観点 | 探すパターン |
|------|-------------|
| コード品質 | 重複コード、不明確な命名、過度な複雑さ、テスト不足 |
| 設計 | 責務の混在、不適切な依存、拡張困難な構造 |
| プロセス | 手戻りの原因、情報不足、判断の迷い |
| ドキュメント | 古い情報、不足、コードとの不整合 |

### 3. 改善提案の導出

特定したパターンから、具体的で実行可能な改善提案を作成する。

**提案の基準：**

- **具体的** —「コードを改善する」ではなく「UserService の validate メソッドを抽出する」
- **実行可能** — 次のタスクまたは近い将来に実施できる粒度
- **効果が明確** — 何がどう良くなるか説明できる

**上位3件に絞る。** 量より質。

## 出力フォーマット

タスクログの「改善提案」セクションに記載する：

```markdown
## 改善提案

### コードベース
- [対象ファイル/モジュール]: [具体的な改善内容] — 理由: [このタスクで観察した根拠]

### プロセス
- [具体的な改善内容] — 理由: [このタスクで観察した根拠]

### 優先度
1. [最も効果が高い提案とその理由]
2. [次に効果が高い提案]
3. [3番目]
```

## Common Mistakes

| ミス | 対策 |
|------|------|
| 一般論を書く（「テストを増やすべき」） | このタスクで具体的に何が不足していたかを示す |
| 改善提案が多すぎる | 上位3件に絞る |
| 実行不可能な提案（「アーキテクチャを刷新」） | 次のタスクで実施できる粒度にする |
| 問題の指摘だけで改善策がない | 必ず具体的な改善策をセットで示す |
| 問題がないのに無理に提案する | 問題がなければ「特になし」でよい |
