# デバッグスキル

バグの調査、原因の特定、修正を体系的に行います。

## デバッグの手順

### ステップ1: 問題の理解

問題を正確に理解するために以下を確認してください：

| 確認項目 | 質問 |
|---------|------|
| **症状** | 具体的に何が起きているか？（エラーメッセージ、異常な動作） |
| **期待動作** | 本来どう動くべきか？ |
| **再現手順** | どのような操作で問題が発生するか？ |
| **発生条件** | 常に発生するか、特定の条件でのみ発生するか？ |
| **発生時期** | いつから発生しているか？（最近の変更と関連するか） |
| **影響範囲** | 他の機能にも影響しているか？ |

### ステップ2: 情報の収集

以下の情報源から関連する情報を収集してください：

1. **エラーメッセージ・スタックトレース**: エラーの詳細を確認
2. **ログ**: アプリケーションログ、サーバーログを確認
3. **最近の変更**: `git log` で最近の変更を確認し、原因となる変更を特定
4. **関連するテスト**: テストが失敗しているか、テストがカバーしていない箇所か
5. **設定ファイル**: 設定の変更が原因でないか確認
6. **依存ライブラリ**: 依存の更新が原因でないか確認

### ステップ3: 原因の切り分け

以下の手法で原因を切り分けてください：

#### 二分探索法

問題の原因となるコードの範囲を狭めていきます：

```
問題が発生するコード全体
  ↓ 半分に分けて、どちらで問題が発生するか確認
上半分 or 下半分
  ↓ さらに半分に分けて確認
問題の原因となる箇所を特定
```

#### レイヤー別切り分け

```
フロントエンド → ネットワーク → バックエンドAPI → ビジネスロジック → データアクセス → データベース
```

各レイヤーの境界で入出力を確認し、問題が発生するレイヤーを特定します。

#### 変更履歴による切り分け

```
git log --oneline -20  # 最近の変更を確認
git bisect start       # 二分探索で原因コミットを特定
```

### ステップ4: 根本原因の分析

原因箇所を特定したら、以下の「5 Whys」手法で根本原因を分析してください：

```
1. なぜエラーが発生したか？→ [直接的な原因]
2. なぜそのコードが問題か？→ [設計/実装上の理由]
3. なぜその設計/実装になったか？→ [背景や制約]
4. なぜそれが見逃されたか？→ [テストやレビューの問題]
5. なぜテスト/レビューで検出できなかったか？→ [プロセス上の問題]
```

全てのWhyに答える必要はありませんが、直接原因だけでなく、可能な範囲で根本原因を特定してください。

### ステップ5: 修正の実装

#### 修正方針の決定

| 修正の種類 | 適用場面 | 注意点 |
|-----------|---------|--------|
| **根本修正** | 根本原因を解消する修正 | 影響範囲を慎重に確認 |
| **対症療法** | 症状を抑える修正 | 一時的な対応。根本修正は別途必要。 |
| **回避策** | 問題を回避する代替実装 | 根本修正が困難な場合の選択肢 |

#### 修正時のチェックリスト

- [ ] 修正が問題の根本原因に対処しているか
- [ ] 修正が他の機能を壊していないか
- [ ] 同様の問題が他の箇所でも発生していないか
- [ ] エッジケースでの動作は正しいか
- [ ] 修正前に問題を再現するテストを書いたか

### ステップ6: 修正の検証

1. **再現テスト**: 問題が再現しなくなったことを確認
2. **リグレッションテスト**: 既存テストが全て通過することを確認
3. **エッジケース確認**: 境界値や異常系での動作を確認
4. **関連機能の確認**: 影響を受ける可能性がある機能の動作確認

### ステップ7: デバッグログの記録

```markdown
# デバッグログ

## 問題の概要
- 症状: [具体的な症状]
- 再現手順: [手順]
- 影響範囲: [影響を受ける機能]

## 調査の過程
1. [調査ステップ1]: [結果]
2. [調査ステップ2]: [結果]
3. [調査ステップ3]: [結果]

## 根本原因
- **直接原因**: [何が問題だったか]
- **根本原因**: [なぜその問題が発生したか]
- **原因箇所**: [ファイルパス:行番号]

## 修正内容
- **修正方針**: [根本修正/対症療法/回避策]
- **変更ファイル**: [ファイルパスと変更概要]
- **追加したテスト**: [テストファイルパスとテスト概要]

## 再発防止
- [この問題の再発を防ぐための提案]
```

## デバッグのベストプラクティス

- **仮説を立ててから検証する**: 闇雲にコードを変更しない
- **1つずつ変更する**: 複数の変更を同時に行うと原因の特定が困難になる
- **再現テストを先に書く**: 修正前に問題を再現するテストを書く
- **ログを活用する**: 適切な箇所にログを追加して動作を追跡する
- **変更を元に戻せるようにする**: git stash や別ブランチを活用する
