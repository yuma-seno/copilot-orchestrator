# テスト戦略スキル

テストの計画立案、テストコードの作成、テスト実行、結果分析を行います。

## テスト計画の手順

### ステップ1: テスト対象の分析

テスト対象のコードを分析し、以下を特定してください：

1. **公開インターフェース**: テスト対象が外部に公開している関数・メソッド・API
2. **入力パターン**: 受け取り可能な入力の範囲
3. **出力パターン**: 期待される出力の範囲
4. **副作用**: ファイルI/O、DB操作、外部API呼び出し等
5. **依存関係**: モック/スタブが必要な外部依存
6. **エラーケース**: 発生しうるエラーの種類

### ステップ2: テスト種別の選択

| テスト種別 | 対象 | 目的 | 実行速度 |
|-----------|------|------|---------|
| **単体テスト** | 関数・メソッド単体 | ロジックの正確性検証 | 高速 |
| **結合テスト** | モジュール間の連携 | コンポーネント間の連携検証 | 中速 |
| **E2Eテスト** | システム全体 | ユーザ視点での動作検証 | 低速 |
| **スナップショットテスト** | UI出力 | UI変更の検出 | 高速 |
| **プロパティベーステスト** | ランダム入力 | 不変条件の検証 | 中速 |

**選択の優先順位：**
1. まず単体テストでロジックの正確性を保証
2. 必要に応じて結合テストでモジュール間の連携を検証
3. 重要なユーザフローにE2Eテストを追加

### ステップ3: テストケースの設計

以下の観点からテストケースを設計してください：

#### 正常系テスト

| テストパターン | 説明 | 例 |
|-------------|------|-----|
| 基本動作 | 最も一般的な入力での正しい動作 | 有効なメールアドレスでログイン |
| 複数の正常入力 | 異なる有効な入力パターン | 異なるフォーマットのメールアドレス |
| 正常系のバリエーション | オプション引数、フラグの組み合わせ | オプション付きでの呼び出し |

#### 異常系テスト

| テストパターン | 説明 | 例 |
|-------------|------|-----|
| 無効な入力 | バリデーションエラーになるべき入力 | 空文字列、null、不正な形式 |
| 存在しないリソース | 存在しないIDやパスへのアクセス | 存在しないユーザIDでの検索 |
| 権限エラー | アクセス権のないリソースへのアクセス | 他ユーザのデータ参照 |
| 外部依存の失敗 | 外部サービスのエラー、タイムアウト | API呼び出し失敗時の挙動 |

#### 境界値テスト

| テストパターン | 説明 | 例 |
|-------------|------|-----|
| 最小値 | 許容される最小の入力 | 長さ1の文字列、値0 |
| 最大値 | 許容される最大の入力 | 最大長の文字列、MAX_INT |
| 境界直前 | 境界のすぐ手前 | 制限値-1 |
| 境界直後 | 境界のすぐ後 | 制限値+1 |
| 空集合 | 空のコレクション | 空配列、空オブジェクト |

### ステップ4: テストコードの実装

#### テストの構造

既存のテストファイルのスタイルに合わせてください。一般的な構造は：

```
Arrange（準備）→ Act（実行）→ Assert（検証）
```

#### テストの命名規則

テスト名から「何をテストしているか」が明確にわかるようにしてください：

| パターン | 例 |
|---------|-----|
| `[メソッド名]_[条件]_[期待結果]` | `login_withValidCredentials_returnsToken` |
| `should [期待動作] when [条件]` | `should return token when credentials are valid` |
| `[条件]の場合、[期待結果]` | `有効な認証情報の場合、トークンが返される` |

#### モック/スタブの使用方針

| 対象 | モックすべきか | 理由 |
|------|--------------|------|
| 外部API呼び出し | ✅ はい | テストの安定性、速度 |
| データベースアクセス | ✅ はい（単体テスト） | テストの独立性 |
| ファイルI/O | ✅ はい（単体テスト） | テストの独立性 |
| 内部のビジネスロジック | ❌ いいえ | 実際のロジックをテストすべき |
| 設定値・定数 | ❌ いいえ | 実際の値でテストすべき |

### ステップ5: テストの実行

1. **新規テストの実行**: 作成したテストが全て通過することを確認
2. **既存テストの実行**: リグレッションがないことを確認
3. **カバレッジの確認**: プロジェクトにカバレッジツールがあれば確認

### ステップ6: テスト結果の記録

```markdown
# テスト結果レポート

## テストサマリ
- テスト実行日時: [ISO8601]
- 総テスト数: [数]
- 成功: [数] / 失敗: [数] / スキップ: [数]
- カバレッジ: [%]（利用可能な場合）

## 新規テスト一覧
| テスト名 | 種別 | テスト対象 | 結果 |
|---------|------|-----------|------|
| [名前] | 単体/結合/E2E | [対象の関数/機能] | ✅/❌ |

## 失敗したテスト（ある場合）
| テスト名 | 失敗理由 | 影響範囲 | 対応方針 |
|---------|---------|---------|---------|
| [名前] | [理由] | [影響] | [方針] |

## カバレッジ詳細（利用可能な場合）
| ファイル | カバレッジ | 未カバー箇所 |
|---------|-----------|------------|
| [パス] | [%] | [行番号や関数名] |
```

## テストの品質基準

- 各テストは独立して実行可能であること（他のテストに依存しない）
- テストは決定的であること（実行するたびに同じ結果になる）
- テスト名から何をテストしているかが明確にわかること
- テストデータはテスト内で完結すること（外部の状態に依存しない）
- テストの実行速度を意識すること（不要なスリープやI/Oを避ける）
