# リファクタリングスキル

外部から見た振る舞いを変えずに、コードの内部構造を改善します。

## リファクタリングの原則

1. **振る舞いを変えない**: リファクタリング前後で外部から見た動作が同じであること
2. **小さなステップ**: 一度に大きな変更をせず、小さな変更を積み重ねる
3. **テストの保証**: 各ステップでテストが通ることを確認する
4. **1つずつ**: リファクタリングと機能変更を同時に行わない

## 手順

### ステップ1: コードの臭いの検出

以下の「コードの臭い」パターンに該当する箇所を特定してください：

#### 膨張（Bloaters）

| 臭い | 症状 | 判定基準 |
|------|------|---------|
| 長いメソッド | 1つの関数が長すぎる | 画面に収まらない、複数の処理を含む |
| 巨大なクラス | 1つのクラスが多くの責務を持つ | フィールドやメソッドが多すぎる |
| 長い引数リスト | 関数の引数が多すぎる | 4つ以上の引数 |
| データの塊 | 常にセットで使われるデータ群 | 同じ変数群が複数箇所に登場 |
| プリミティブの偏愛 | プリミティブ型の過度な使用 | 文字列でドメイン概念を表現 |

#### 変更の困難さ（Change Preventers）

| 臭い | 症状 | 判定基準 |
|------|------|---------|
| 変更の分散 | 1つの変更が複数箇所に影響 | 1つの機能変更で多くのファイルを修正 |
| 変更の集中 | 多くの変更が1つのクラスに集中 | 異なる理由での変更が同じクラスに集中 |
| 並行継承 | サブクラスの追加がセットで必要 | クラス階層が複雑化 |

#### 不要な複雑さ（Dispensables）

| 臭い | 症状 | 判定基準 |
|------|------|---------|
| 重複コード | 同じロジックの繰り返し | コピー&ペーストされたコード |
| 死んだコード | 使われていないコード | 未使用の変数、関数、クラス |
| 投機的な汎用性 | 将来のための過剰な抽象化 | 使われていないインターフェースや設定 |
| コメントの過剰 | コードの代わりのコメント | コードの可読性を上げれば不要なコメント |

### ステップ2: リファクタリング計画

特定した問題の優先度を決め、リファクタリングの順序を計画してください：

**優先度の判断基準：**

| 優先度 | 基準 |
|--------|------|
| 高 | 今後の開発を大きく妨げている。変更のたびにバグが発生する。 |
| 中 | 可読性が低く理解に時間がかかる。重複コードが存在する。 |
| 低 | 改善の余地はあるが、実害は少ない。 |

### ステップ3: リファクタリングパターンの適用

#### よく使うリファクタリングパターン

| パターン | 適用場面 | 手順概要 |
|---------|---------|---------|
| **関数の抽出** | 長い関数を分割 | 関連する処理をまとめて新しい関数に切り出す |
| **変数の抽出** | 複雑な式の簡略化 | 複雑な式を意味のある名前の変数に格納 |
| **インライン化** | 不要な間接参照の除去 | 呼び出し先の処理が簡単で、間接参照が不要な場合に統合 |
| **関数の移動** | 適切なモジュールへの移動 | 関数をより適切なクラス/モジュールに移動 |
| **パラメータオブジェクト** | 引数リストの短縮 | 関連する引数をオブジェクトにまとめる |
| **ガード節** | ネストの削減 | 早期リターンでネストを浅くする |
| **ポリモーフィズム** | 条件分岐の簡略化 | switch/if文をポリモーフィズムで置き換える |
| **ストラテジパターン** | アルゴリズムの切り替え | アルゴリズムをオブジェクトとして分離 |

### ステップ4: 段階的な改善の実施

1. **テストの確認**: リファクタリング対象のコードに十分なテストがあるか確認
   - テストが不足している場合は先にテストを追加する
2. **1つのパターンを適用**: 小さな変更を行う
3. **テストの実行**: 全テストが通ることを確認
4. **次のパターンを適用**: テストが通った場合のみ次に進む
5. **繰り返し**: 改善が完了するまで2〜4を繰り返す

### ステップ5: リファクタリングログの記録

```markdown
# リファクタリングログ

## 対象
- ファイル: [ファイルパス]
- 検出した問題: [コードの臭いの種類]

## 適用したパターン
| # | パターン | 対象 | 変更内容 |
|---|---------|------|---------|
| 1 | [パターン名] | [対象の関数/クラス] | [変更概要] |

## テスト結果
- 変更前: 全テスト通過 ✅
- 変更後: 全テスト通過 ✅

## 改善効果
- [改善されたポイント（可読性、保守性、テスタビリティ等）]
```

## 注意事項

- リファクタリングと機能変更を同時に行わない
- テストなしにリファクタリングを開始しない
- パフォーマンスの最適化は計測に基づいて行う（推測で最適化しない）
- 過度な抽象化を避ける（YAGNI: You Aren't Gonna Need It）
