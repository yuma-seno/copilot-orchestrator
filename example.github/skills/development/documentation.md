# ドキュメント作成スキル

技術ドキュメントの作成・更新を行います。

## ドキュメントタイプ別ガイド

### README.md

プロジェクトの入り口となるドキュメントです。

**含めるべきセクション：**

```markdown
# プロジェクト名

[1〜2文のプロジェクト概要]

## 概要
[プロジェクトの目的と主要な機能の説明]

## 前提条件
[実行に必要な環境・ツールのバージョン]

## セットアップ
[インストール手順をコマンド例付きで]

## 使い方
[基本的な使い方をコード例付きで]

## ディレクトリ構成
[主要なディレクトリとファイルの説明]

## 設定
[設定ファイル、環境変数の説明]

## 開発
[開発環境のセットアップ、ビルド、テスト手順]

## ライセンス
[ライセンス情報]
```

### API ドキュメント

API のエンドポイント仕様を記述します。

**各エンドポイントに含めるべき情報：**

```markdown
### [メソッド] [パス]

[エンドポイントの概要説明]

**リクエスト:**

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| [名前] | [型] | [はい/いいえ] | [説明] |

**リクエスト例:**
[コード例]

**レスポンス:**

| フィールド | 型 | 説明 |
|-----------|-----|------|
| [名前] | [型] | [説明] |

**レスポンス例:**
[コード例]

**エラーレスポンス:**

| ステータスコード | 説明 | 対処法 |
|----------------|------|--------|
| [コード] | [説明] | [対処法] |
```

### 設計書

技術的な設計を記述します。（`skills/development/architecture-design.md` の成果物）

### 手順書

操作手順やプロセスを記述します。

**構成の原則：**
- 各手順は明確な番号付きリストにする
- 各手順の結果（期待される出力）を明示する
- コマンド例は実行可能な形で記載する
- 注意事項やトラブルシューティングを含める

### CHANGELOG

変更履歴を記録します。

**フォーマット（Keep a Changelog 形式）：**

```markdown
# Changelog

## [Unreleased]

### Added
- [追加された機能の説明]

### Changed
- [変更された機能の説明]

### Fixed
- [修正されたバグの説明]

### Removed
- [削除された機能の説明]
```

## ドキュメント作成の手順

### ステップ1: 対象の理解

1. ドキュメント対象のコード・機能を理解する
2. 対象読者を特定する（エンドユーザ、開発者、運用担当者）
3. 既存のドキュメントがあれば確認する

### ステップ2: 構成の設計

1. ドキュメントタイプに応じた構成テンプレートを選択
2. 対象読者に必要な情報を洗い出す
3. セクションの順序を決定（重要なものから）

### ステップ3: 執筆

以下の原則に従って執筆してください：

| 原則 | 具体的な行動 |
|------|------------|
| **正確性** | コードの実際の動作に基づいて記述する。推測で書かない。 |
| **簡潔性** | 不必要に長い説明を避ける。1文は短く。 |
| **具体性** | 抽象的な説明より、コード例や具体的な手順を示す。 |
| **一貫性** | 用語、書式、スタイルを統一する。 |
| **最新性** | コードとドキュメントの内容が一致していることを確認する。 |

### ステップ4: レビューと確認

- コード例が正しく動作することを確認
- リンクが有効であることを確認
- 用語の一貫性を確認
- 対象読者の視点で読み返す

## 執筆ガイドライン

### やるべきこと

- コード例は実行可能な状態で記載する
- 手順書には期待される結果を明示する
- 図表を活用して複雑な概念を説明する
- 変更があったらドキュメントも更新する

### やってはいけないこと

- コードと矛盾する内容を書く
- 内部実装の詳細を外部向けドキュメントに書く
- 専門用語を説明なしに使う（対象読者に応じて）
- TODO やプレースホルダーを残したまま公開する
