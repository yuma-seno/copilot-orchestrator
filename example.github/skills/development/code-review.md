# コードレビュースキル

コードの品質・正確性・セキュリティ・保守性を評価し、具体的な指摘と改善提案を行います。

## レビューの手順

### ステップ1: レビュー準備

1. **変更の目的を理解**: 要件定義書、設計書、またはPR説明文を確認
2. **変更ファイルの特定**: 変更されたファイルの一覧を取得
3. **変更の規模を把握**: 変更行数、影響ファイル数を確認

### ステップ2: レビュー観点別チェック

#### 観点1: コード品質

| チェック項目 | 確認ポイント |
|------------|------------|
| 命名の適切性 | 変数名・関数名・クラス名から目的が読み取れるか。プロジェクトの命名規約に準拠しているか。 |
| 関数の長さ | 1つの関数が長すぎないか（目安: 画面に収まる程度）。分割すべきか。 |
| 単一責任 | 1つの関数/クラスが1つの責務のみを持っているか。 |
| 重複コード | コピー&ペーストされたコードがないか。共通化すべきか。 |
| 不要なコード | 未使用の変数、インポート、関数がないか。コメントアウトされたコードが残っていないか。 |
| コーディング規約 | プロジェクトの規約（インデント、セミコロン、引用符等）に準拠しているか。 |
| 一貫性 | 同じプロジェクト内で一貫したスタイルが保たれているか。 |

#### 観点2: ロジックの正確性

| チェック項目 | 確認ポイント |
|------------|------------|
| 要件の充足 | 変更が要件を正しく満たしているか。 |
| エッジケース | null/undefined、空文字列、空配列、境界値の処理が適切か。 |
| 型安全性 | 型の不整合がないか。暗黙の型変換による問題がないか。 |
| 条件分岐 | if/else、switch文のすべての分岐が意図通りか。条件の漏れがないか。 |
| ループ | 無限ループの可能性がないか。ループの終了条件が正しいか。 |
| 非同期処理 | async/await、Promise の処理が正しいか。エラーが適切にハンドリングされているか。 |
| 並行性 | レースコンディションの可能性がないか。 |

#### 観点3: セキュリティ

| チェック項目 | 確認ポイント | 重要度 |
|------------|------------|--------|
| 入力バリデーション | ユーザ入力が適切に検証・サニタイズされているか | 高 |
| SQLインジェクション | パラメータ化クエリが使用されているか | 高 |
| XSS | HTMLエスケープ処理が行われているか | 高 |
| CSRF | CSRF対策トークンが使用されているか | 高 |
| 認証・認可 | 適切なアクセス制御が行われているか | 高 |
| 機密情報 | パスワード、APIキー等がハードコードされていないか | 高 |
| ファイルアクセス | パストラバーサルの危険がないか | 中 |
| 依存ライブラリ | 既知の脆弱性がある依存がないか | 中 |
| ログ出力 | 機密情報がログに含まれていないか | 中 |

#### 観点4: パフォーマンス

| チェック項目 | 確認ポイント |
|------------|------------|
| アルゴリズム効率 | O(n²) 以上の計算量が不必要に使われていないか |
| メモリ使用 | 大量のデータをメモリに保持していないか |
| N+1問題 | ループ内でDBクエリやAPI呼び出しを行っていないか |
| 不必要な処理 | 同じ計算を繰り返していないか。キャッシュすべきか。 |
| 非同期の活用 | 並列実行可能な処理が直列で実行されていないか |

#### 観点5: 保守性

| チェック項目 | 確認ポイント |
|------------|------------|
| 可読性 | 他の開発者がこのコードを理解できるか |
| テスタビリティ | テストを書きやすい構造か。依存の注入が可能か。 |
| 拡張性 | 将来の要件変更に対して柔軟に対応できる構造か |
| ドキュメント | 複雑なロジックにコメントが付けられているか |
| エラーメッセージ | エラーメッセージが問題の特定に役立つか |

### ステップ3: 指摘の分類

指摘を以下の3段階に分類してください：

| 分類 | 基準 | 対応 |
|------|------|------|
| **CRITICAL（重大）** | セキュリティ脆弱性、データ損失の危険、機能的なバグ | 修正必須。マージ前に対応が必要。 |
| **WARNING（警告）** | コード品質の問題、パフォーマンスの懸念、テスト不足 | 修正推奨。可能な限り対応。 |
| **INFO（情報）** | スタイルの改善提案、代替実装の提案、ドキュメントの補足 | 検討推奨。対応は任意。 |

### ステップ4: レビューレポートの作成

`artifacts/review-report.md` に以下のテンプレートでレポートを出力してください：

```markdown
# レビューレポート

## サマリ
- レビュー対象: [ファイルリスト]
- 総合評価: [APPROVE / REQUEST_CHANGES / COMMENT]
- 指摘数: CRITICAL: [数] / WARNING: [数] / INFO: [数]

## 指摘事項

### CRITICAL（修正必須）

#### C-001: [指摘タイトル]
- **ファイル**: [ファイルパス:行番号]
- **問題**: [何が問題か]
- **理由**: [なぜ問題か（影響、リスク）]
- **修正案**:
  ```[言語]
  // 修正後のコード例
  ```

### WARNING（修正推奨）

#### W-001: [指摘タイトル]
- **ファイル**: [ファイルパス:行番号]
- **問題**: [何が問題か]
- **修正案**: [推奨する修正方法]

### INFO（検討推奨）

#### I-001: [提案タイトル]
- **ファイル**: [ファイルパス:行番号]
- **提案**: [提案内容]

## 良い点
- [良い実装や設計について具体的に言及]
```

## レビューのルール

- **客観的に**: 個人の好みではなく、品質・安全性・保守性の観点で判断する
- **具体的に**: 「何が問題か」「なぜ問題か」「どう修正すべきか」を明示する
- **建設的に**: 問題点だけでなく、良い点も言及する
- **コード例を示す**: 修正案には可能な限りコード例を添える
