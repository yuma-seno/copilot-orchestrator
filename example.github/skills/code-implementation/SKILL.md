# コード実装スキル

設計に基づいてコードの実装を行います。

## 実装の手順

### ステップ1: 実装準備

実装を開始する前に、以下を確認してください：

1. **設計の確認**: `artifacts/design.md` またはユーザからの指示で設計方針を理解
2. **コーディング規約の把握**:
   - リンター設定（`.eslintrc`, `.prettierrc`, `pyproject.toml` 等）を確認
   - 既存コードのスタイルを確認（命名規則、インデント、コメントスタイル）
   - インポート/エクスポートの方式を確認
3. **関連コードの理解**:
   - 変更対象のファイルとその周辺コードを読む
   - 依存関係を把握する
   - 似た機能の既存実装があれば参考にする

### ステップ2: 実装計画

大きな実装は以下のように小さなステップに分割してください：

```markdown
### 実装ステップ
1. [ステップ1の内容]（例: 型定義・インターフェースの作成）
2. [ステップ2の内容]（例: コアロジックの実装）
3. [ステップ3の内容]（例: エラーハンドリングの追加）
4. [ステップ4の内容]（例: 既存コードとの統合）
```

**ステップ分割の原則：**
- 各ステップの後に基本動作が確認できる状態にする
- 依存関係の順序で実装する（インターフェース → 実装 → 統合）
- 既存のテストを壊さない単位で変更する

### ステップ3: コードの実装

以下の原則に従って実装してください：

#### コーディング原則

| 原則 | 具体的な行動 |
|------|------------|
| **既存規約の遵守** | プロジェクトのコーディングスタイルに合わせる。新しいスタイルを持ち込まない。 |
| **命名の明確性** | 変数名・関数名から目的が読み取れるようにする。既存の命名パターンに合わせる。 |
| **単一責任** | 1つの関数は1つのことだけを行う。関数が長くなったら分割を検討。 |
| **早期リターン** | ガード節でエラーケースを先に処理し、正常パスのネストを浅くする。 |
| **不変性の優先** | 可能な限り `const`/`readonly`/不変データ構造を使用する。 |
| **明示的なエラーハンドリング** | エラーを握りつぶさない。適切にキャッチして処理 or 上位に伝播する。 |

#### セキュリティチェックリスト

実装中に以下を確認してください：

- [ ] ユーザ入力は適切にバリデーション・サニタイズされているか
- [ ] SQLインジェクション、XSS、CSRF等の脆弱性がないか
- [ ] 機密情報（パスワード、APIキー）がハードコードされていないか
- [ ] ファイルパスの操作にパストラバーサルの危険がないか
- [ ] 適切な権限チェックが行われているか

#### エラーハンドリングパターン

```
エラーの発生
  ↓
エラーの種類を判別
  ├─ 回復可能なエラー → リトライ or フォールバック処理
  ├─ バリデーションエラー → ユーザにわかりやすいエラーメッセージを返す
  ├─ システムエラー → ログに記録し、適切なエラーレスポンスを返す
  └─ 予期しないエラー → ログに記録し、安全に失敗する（クラッシュさせない）
```

### ステップ4: 基本動作確認

実装後、以下の確認を行ってください：

1. **構文チェック**: ビルド/コンパイルが通ることを確認
   - TypeScript: `tsc --noEmit`
   - Go: `go build ./...`
   - Python: `python -m py_compile`
   - 等、プロジェクトのビルドコマンドを使用

2. **リンターチェック**: リンターの警告・エラーがないことを確認
   - プロジェクトの lint コマンドがあれば実行

3. **既存テスト**: 既存テストが壊れていないことを確認
   - プロジェクトのテストコマンドを実行

4. **動作確認**: 主要な動作パスが機能することを確認

### ステップ5: 実装ログの記録

実装内容を以下の形式で記録してください：

```markdown
# 実装ログ

## 変更ファイル一覧
| ファイルパス | 変更種別 | 変更概要 |
|------------|---------|---------|
| [パス] | 新規/変更/削除 | [概要] |

## 実装のポイント
- [重要な実装判断とその理由]
- [採用した設計パターンとその理由]

## 確認済み事項
- [ ] ビルド/コンパイル通過
- [ ] リンター通過
- [ ] 既存テスト通過
- [ ] 基本動作確認

## テスト時に確認すべきポイント
- [テストフェーズで特に確認すべき項目]
- [エッジケースや境界値の確認ポイント]
```

## やってはいけないこと

- **既存テストの削除・無効化**: テストが壊れる場合は実装を見直す
- **不要なコードの追加**: 使わないコード、将来のための過剰な抽象化を避ける
- **スタイルの混在**: プロジェクトと異なるコーディングスタイルを持ち込まない
- **デバッグコードの残留**: `console.log`、`print`、TODO コメント等を本番コードに残さない
- **セキュリティの軽視**: 入力バリデーション、エラーハンドリングを省略しない
