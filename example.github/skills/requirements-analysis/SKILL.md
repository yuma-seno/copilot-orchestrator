# Requirements Analysis

ユーザの指示から明確な要件を導出する。曖昧さを排除し、検証可能な受け入れ基準を定義する。

## 分析フレームワーク

以下を明確にする。不明な項目は推測せず質問事項として記録する。

| 項目 | 問い |
|------|------|
| WHO | 誰がこの機能を使うか |
| WHAT | 何ができるようになるか |
| WHY | なぜ必要か（解決する問題は何か） |
| WHERE | どのファイル・モジュール・レイヤが対象か |
| HOW | 入力 → 処理 → 出力の流れ |

## 受け入れ基準

各要件に対して、客観的に判定可能な受け入れ基準を定義する。

<Good>
「ユーザがメールアドレスとパスワードを入力してログインボタンを押すと、JWTトークンが返される。メールが未登録の場合は 404 が返る。パスワードが誤りの場合は 401 が返る」
</Good>

<Bad>
「ユーザがログインできる」
</Bad>

**基準:** 受け入れ基準を読んだ第三者が、テストコードを書けるか？書けないなら不十分。

## エッジケースチェック

要件ごとに以下のカテゴリでエッジケースを洗い出す：

| カテゴリ | 確認項目 |
|---------|---------|
| 入力の境界 | null, 空文字, 空配列, 最大長, 最小値, 最大値, 負の数 |
| 状態の前提 | 未認証, 権限不足, データ不存在, 重複 |
| 並行性 | 同時実行, レースコンディション |
| 外部依存 | ネットワークエラー, タイムアウト, API変更 |

## 要件定義書

`artifacts/requirements.md` に出力する：

```markdown
# 要件定義書

## 目的
[解決する問題と達成したいこと]

## スコープ
- 対象: [...]
- 対象外: [...]

## 機能要件
| ID | 要件 | 受け入れ基準 |
|----|------|-------------|
| FR-001 | [要件] | [検証可能な基準] |

## 非機能要件
[パフォーマンス、セキュリティ、互換性等]

## エッジケース
[洗い出した境界条件]

## 未解決事項
[推測せずに記録した質問事項]
```

## Common Mistakes

| ミス | 対策 |
|------|------|
| 受け入れ基準が曖昧 | 「〜できる」ではなく入力・操作・期待結果を具体的に |
| 実装方法に踏み込む | それは設計フェーズの責務 |
| 不明点を推測で埋める | 質問事項として明示する |
