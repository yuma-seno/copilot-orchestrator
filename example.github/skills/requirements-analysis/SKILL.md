# 要件分析スキル

ユーザの指示やプロジェクトのコンテキストから、明確な要件を定義します。

## 分析の手順

### ステップ1: 現状の把握

作業対象のプロジェクトについて以下を調査してください：

1. **技術スタック**: 使用している言語、フレームワーク、ライブラリ
   - `package.json`, `requirements.txt`, `go.mod`, `Cargo.toml` 等の依存定義ファイルを確認
2. **プロジェクト構造**: ディレクトリ構成、モジュール分割の方針
   - ルートディレクトリとその直下のファイル・フォルダを確認
3. **既存のドキュメント**: README、APIドキュメント、設計書
   - `README.md`, `docs/`, `CONTRIBUTING.md` 等を確認
4. **テスト構成**: テストフレームワーク、テストディレクトリ構成
   - テスト関連の設定ファイル、テストディレクトリを確認
5. **設定ファイル**: CI/CD、リンター、フォーマッター
   - `.github/workflows/`, `.eslintrc`, `.prettierrc`, `tsconfig.json` 等を確認

### ステップ2: 指示の構造化

ユーザの指示を以下のフレームワークで整理してください：

#### WHO（誰が）
- この機能を使うのは誰か（エンドユーザ、開発者、管理者）
- 関連するステークホルダーは誰か

#### WHAT（何を）
- 具体的に何を実現するか
- どのような入力を受け、どのような出力を返すか
- どのような状態変化が起きるか

#### WHY（なぜ）
- なぜこの変更が必要か（ビジネス背景、技術的背景）
- 解決したい課題は何か

#### WHERE（どこで）
- どのモジュール・コンポーネントに影響するか
- どのファイルが変更対象か

#### HOW（どのように）※概要レベル
- 技術的なアプローチの概要（詳細は設計フェーズで）
- 既存の類似実装があるか

### ステップ3: 要件の分類と整理

#### 機能要件（Functional Requirements）

システムが提供すべき具体的な機能を列挙します：

```markdown
### 機能要件

| ID | 要件 | 詳細 | 受け入れ基準 |
|----|------|------|-------------|
| FR-001 | [要件名] | [具体的な説明] | [客観的に判定可能な基準] |
| FR-002 | [要件名] | [具体的な説明] | [客観的に判定可能な基準] |
```

**良い受け入れ基準の例：**
- ✅ 「ユーザがメールアドレスとパスワードを入力してログインボタンを押すと、トークンが返される」
- ❌ 「ユーザがログインできる」（曖昧）

#### 非機能要件（Non-Functional Requirements）

| カテゴリ | チェック項目 |
|---------|------------|
| パフォーマンス | レスポンス時間、スループット、リソース使用量 |
| セキュリティ | 認証・認可、データ暗号化、入力バリデーション |
| 可用性 | エラーハンドリング、フォールバック |
| 保守性 | コードの可読性、テスタビリティ、拡張性 |
| 互換性 | 既存APIとの互換性、ブラウザ互換性 |

#### 制約事項（Constraints）

- 技術的制約（使用言語、フレームワーク、ライブラリのバージョン）
- ビジネス制約（既存のAPIインターフェースを変更しない、等）
- 環境制約（対応ブラウザ、Node.jsバージョン、等）

### ステップ4: 不明点の洗い出し

要件に不明確な点がある場合、以下の形式で質問事項をまとめてください：

```markdown
### 確認事項

| # | 質問 | 背景 | デフォルト案 |
|---|------|------|------------|
| 1 | [具体的な質問] | [なぜこの確認が必要か] | [仮の回答案があれば] |
| 2 | [具体的な質問] | [なぜこの確認が必要か] | [仮の回答案があれば] |
```

### ステップ5: 要件定義書の出力

以下のテンプレートで `artifacts/requirements.md` を作成してください：

```markdown
# 要件定義書

## 1. 背景・目的
[なぜこの変更が必要か、解決する課題]

## 2. スコープ
### 対象
- [対象のファイル/機能/モジュール]

### 対象外
- [明示的にスコープ外とするもの]

## 3. 機能要件
| ID | 要件 | 詳細 | 受け入れ基準 |
|----|------|------|-------------|
| FR-001 | [要件名] | [詳細] | [基準] |

## 4. 非機能要件
| ID | カテゴリ | 要件 | 基準 |
|----|---------|------|------|
| NFR-001 | [カテゴリ] | [要件] | [基準] |

## 5. 制約事項
1. [制約の説明]

## 6. 前提条件
1. [前提の説明]

## 7. 確認事項（未解決）
| # | 質問 | 影響範囲 |
|---|------|---------|
```

## 注意事項

- 受け入れ基準は必ず客観的に判定可能な形で記述する
- 技術的な実装方法には踏み込まない（設計フェーズの責務）
- 不明点を無理に推測せず、質問事項として明示する
- 既存のコードベースの規約・パターンを尊重する
