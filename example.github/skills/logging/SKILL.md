# ログ記録スキル

作業ログの記録フォーマットと規約を定義します。
このスキルはオーケストレーションセッションに限らず、作業内容の記録が必要な場面で汎用的に使用できます。

## ログファイルの命名規則

- **ファイル名形式**: `[yyyymmddhhmmss]-[name].md`
  - `yyyymmddhhmmss`: 作成日時（14桁）
  - `name`: ログの内容を示す短い英語名（ケバブケース）
  - 例: `20260329123456-implement-auth-api.md`

## ログの種類と出力先

### 1. タスクログ

各タスクの実行記録です。

**出力先**: `logs/[timestamp]-[task-name].md`

```markdown
# タスクログ: [タスク名]

## 基本情報
- タスクID: [ID]（例: task-001）
- 開始日時: [ISO8601]
- 完了日時: [ISO8601]
- 結果: [成功 / 失敗 / 要確認]

## 指示内容
[受け取った指示の要約]

## 実行内容
1. [実行した作業1]
   - 結果: [結果の説明]
2. [実行した作業2]
   - 結果: [結果の説明]

## 変更されたファイル
| ファイルパス | 変更種別 | 変更概要 |
|------------|---------|---------|
| [パス] | 新規/変更/削除 | [概要] |

## 判断記録
| 場面 | 判断内容 | 理由 |
|------|---------|------|
| [判断が必要だった場面] | [選択した方針] | [理由] |

## 次のステップへの引き継ぎ
- [次のタスクで注意すべき事項]
- [未解決の課題]
- [前提条件や依存関係]
```

### 2. 判断ログ

重要な技術的判断の記録です。影響が大きい判断や、後から理由を追跡する必要がある判断を記録します。

**出力先**: `logs/decisions/[timestamp]-[decision-name].md`

```markdown
# 判断記録: [判断の概要を一文で]

## コンテキスト
[判断が必要になった背景と制約条件]

## 検討した選択肢

### 選択肢A: [名前]
- **概要**: [選択肢の説明]
- **メリット**: [利点を箇条書きで]
- **デメリット**: [欠点を箇条書きで]
- **リスク**: [リスクがあれば]

### 選択肢B: [名前]
- **概要**: [選択肢の説明]
- **メリット**: [利点を箇条書きで]
- **デメリット**: [欠点を箇条書きで]
- **リスク**: [リスクがあれば]

## 判断結果
- **選択**: [選択した選択肢]
- **理由**: [選択した主な理由]

## 影響範囲
- [この判断が影響するファイル、機能、タスク]

## 見直し条件
- [この判断を見直すべき条件があれば記載]
```

## ログ記録のルール

### 記録すべきこと

| 種別 | 記録対象 |
|------|---------|
| **常に記録** | タスクの開始・完了、変更したファイル、テスト結果 |
| **常に記録** | 技術的な判断とその理由 |
| **常に記録** | エラーの発生と対処内容 |
| **状況に応じて** | 調査の過程で得た知見 |
| **状況に応じて** | 他の選択肢を検討した経緯 |

### 記録してはいけないこと

- パスワード、APIキー、トークン等の機密情報
- 個人情報（メールアドレス、氏名等）
- ライセンスで保護されたコードの全文コピー

### 記録の品質基準

| 基準 | 説明 |
|------|------|
| **簡潔性** | 事実を簡潔に記録する。不要な修飾語を避ける。 |
| **追跡可能性** | 後から経緯を追跡できるように、因果関係を明示する。 |
| **客観性** | 事実と意見を区別する。意見には根拠を添える。 |
| **完全性** | 重要な情報が欠落しないようにする。 |
