# 状態管理スキル

セッションの状態管理とエージェント間のコンテキスト共有を定義します。

## セッション状態の遷移

```
initializing → active → completing → completed
                 ↓
              failed
```

| 状態 | 説明 | 遷移条件 |
|------|------|---------|
| `initializing` | セッション初期化中 | セッションディレクトリ作成開始時 |
| `active` | セッション実行中 | 初期化完了時に遷移 |
| `completing` | 完了処理中 | 全タスク完了時に遷移 |
| `completed` | 正常完了 | 完了処理（サマリ作成等）終了時に遷移 |
| `failed` | 異常終了 | 回復不能なエラー発生時に遷移 |

## session.json の構造

```json
{
  "sessionId": "yyyymmddhhmmssfff",
  "createdAt": "2026-03-29T12:34:56.789Z",
  "status": "active",
  "rules": {
    "questionGranularity": "medium",
    "reviewPolicy": "auto",
    "scope": "src/auth/"
  },
  "tasks": [
    {
      "id": "task-001",
      "name": "認証APIの実装",
      "status": "completed",
      "assignedAt": "2026-03-29T12:35:00Z",
      "completedAt": "2026-03-29T12:40:00Z",
      "summary": "POST /api/auth/login と /api/auth/register を実装完了"
    }
  ],
  "history": [
    {
      "timestamp": "2026-03-29T12:34:56Z",
      "event": "session-init",
      "details": "セッション初期化完了"
    }
  ]
}
```

## 更新ルール

### 更新の責務

| 操作 | 実行者 |
|------|--------|
| session.json の作成 | サブエージェント（オーケストレーターの指示により） |
| status の更新 | サブエージェント（オーケストレーターの指示により） |
| rules の設定 | サブエージェント（オーケストレーターの指示により） |
| tasks の追加・更新 | サブエージェント（タスク完了時に自身で更新） |
| history の追記 | サブエージェント（状態変更時に自身で追記） |
| session.json の参照 | サブエージェント（作業開始時にルール確認のため） |

### 更新時の制約

1. **履歴の不変性**: `history` の既存エントリは編集・削除しない。追記のみ。
2. **ステータスの整合性**: `status` は上記の遷移ルールに従ってのみ変更する。
3. **タスクの更新**: タスクの `status` は `pending → in-progress → completed/failed` の順で変更する。

### イベントタイプ

| イベント | 発火タイミング |
|---------|--------------|
| `session-init` | セッション初期化完了時 |
| `rules-set` | セッションルール設定時 |
| `task-assigned` | タスクがサブエージェントに割り当てられた時 |
| `task-started` | タスクの実行が開始された時 |
| `task-completed` | タスクが正常完了した時 |
| `task-failed` | タスクが失敗した時 |
| `quality-gate-pass` | 品質ゲートを通過した時 |
| `quality-gate-fail` | 品質ゲートで差し戻された時 |
| `flow-changed` | ワークフローが変更された時 |
| `session-completing` | セッション完了処理開始時 |
| `session-completed` | セッション正常完了時 |
| `session-failed` | セッション異常終了時 |

## コンテキストの引き継ぎ

### サブエージェントがコンテキストを取得する手順

サブエージェントが新しいタスクを開始する際は、以下の順序でコンテキストを取得してください：

1. **session.json を読む**
   - `rules` からセッションルール（質問粒度、レビュー方針、スコープ）を確認
   - `tasks` から過去のタスクの結果を確認
   - `history` から重要なイベントを確認

2. **過去のタスクログを読む**
   - `logs/` ディレクトリから、関連する過去のタスクログを確認
   - 特に前のタスクの「次のステップへの引き継ぎ」セクションを確認

3. **判断ログを読む**
   - `logs/decisions/` ディレクトリから、重要な判断を確認
   - 判断の理由を理解し、矛盾する方針を取らないようにする

4. **成果物を読む**
   - `artifacts/` ディレクトリから、関連する成果物（要件定義書、設計書等）を確認

### オーケストレーターへのサマリ返却ルール

サブエージェントからオーケストレーターへは、セッションディレクトリ内の詳細情報ではなく、**判断に必要な最小限のサマリのみ**を返します：

**サマリに含めるべき情報：**
- タスク名と結果（成功/失敗/要確認）
- 変更の概要（1〜3行）
- 次のステップで注意すべき事項
- 詳細ログのパス

**サマリに含めない情報：**
- コードの詳細な差分
- 調査過程の詳細
- 判断の詳細な経緯（ログに記録済み）
