# Refactoring

外部から見た振る舞いを変えずに、コードの内部構造を改善する。

**コア原則:** 振る舞いを変えない。小さなステップで。各ステップでテストが通ることを確認する。

<HARD-GATE>
リファクタリングと機能変更を同時に行わない。テストなしにリファクタリングを開始しない。
</HARD-GATE>

## When to Use

- コードの可読性・保守性を改善するとき
- 重複コードを解消するとき
- 技術的負債を解消するとき

**使わない場面:** 新機能の実装 → code-implementation / バグ修正 → debugging

## チェックリスト

1. **テスト確認** — 対象コードに十分なテストがあるか確認。不足なら先にテストを追加
2. **コードの臭い検出** — 下記のパターンに該当する箇所を特定
3. **優先度決定** — 影響度と改善効果で順序を決める
4. **1つずつ適用** — パターンを1つ適用 → テスト実行 → 通過を確認 → 次へ
5. **完了確認** — 全テスト通過、振る舞い不変

## コードの臭い検出パターン

| カテゴリ | 臭い | 判定基準 |
|---------|------|---------|
| 膨張 | 長いメソッド | 画面に収まらない、複数の処理を含む |
| 膨張 | 巨大なクラス | フィールドやメソッドが多すぎる |
| 膨張 | 長い引数リスト | 4つ以上の引数 |
| 不要な複雑さ | 重複コード | コピー&ペーストされたコード |
| 不要な複雑さ | 死んだコード | 未使用の変数、関数、クラス |
| 不要な複雑さ | 投機的な汎用性 | 使われていないインターフェースや設定 |
| 変更の困難さ | 変更の分散 | 1つの変更で多くのファイルを修正 |

## よく使うパターン

| パターン | 適用場面 |
|---------|---------|
| 関数の抽出 | 長い関数を分割 |
| 変数の抽出 | 複雑な式を意味のある名前に |
| 関数の移動 | 適切なモジュールへ移動 |
| パラメータオブジェクト | 引数リストの短縮 |
| ガード節 | ネストの削減（早期リターン） |

## Common Mistakes

| ミス | 対策 |
|------|------|
| 機能変更とリファクタリングを同時に行う | 別コミットに分ける |
| テストなしにリファクタリング開始 | テストを先に追加する |
| 推測でパフォーマンス最適化 | 計測に基づいて最適化する |
| 過度な抽象化 | YAGNI: 現在不要な抽象化はしない |
