# Systematic Debugging

根本原因の調査なしに修正を試みない。症状の修正は失敗である。

<HARD-GATE>
根本原因の調査（Phase 1）を完了するまで、修正コードを書いてはいけない。
</HARD-GATE>

## When to Use

あらゆる技術的問題に使用する：

- テスト失敗、バグ、予期しない動作
- パフォーマンス問題、ビルド失敗
- 時間のプレッシャーがあるとき（**特に**このとき。急ぐほど体系的でなければならない）
- 「ちょっとした修正で直りそう」と思ったとき（**特に**このとき）

## 4つのフェーズ

各フェーズは前のフェーズが完了するまで進めない。

### Phase 1: 根本原因の調査

**修正を試みる前に必ず実行する：**

1. **エラーメッセージを注意深く読む** — 行番号、スタックトレース、エラーコードを全て確認
2. **一貫して再現する** — 再現手順と発生条件を特定。再現できないなら推測しない
3. **最近の変更を確認** — `git log`, `git diff`。設定変更、依存更新も対象
4. **データフローを追跡** — 不正な値がどこで生じたか、呼び出し元を順に遡る

### Phase 2: パターン分析

1. **同じコードベースの動作する例を見つける** — 壊れているものと何が違うか
2. **リファレンス実装があれば完全に読む** — 流し読みしない
3. **全ての違いを列挙する** — 「これは関係ない」と思い込まない

### Phase 3: 仮説と検証

1. **仮説を1つ立てる** — 「原因はXだと考える。なぜならY」
2. **最小の変更でテストする** — 一度に1つの変数だけ変更
3. **結果を検証** — 通った → Phase 4。通らない → 新しい仮説を立て直す。**上から修正を重ねない**

### Phase 4: 修正の実装

1. **失敗するテストケースを先に作る** — 問題を再現するテスト
2. **1つの修正を実装する** — 根本原因に対処。"ついでに"の改善はしない
3. **検証する** — テスト通過、他のテストも通過、問題が解消

**3回修正しても直らない場合：** 停止する。アーキテクチャの問題を疑い、ユーザと相談する。

## Red Flags — 今すぐ止まって Phase 1 に戻る

以下の思考が浮かんだら、プロセスに違反している：

| 思考 | 現実 |
|------|------|
| 「とりあえずXを変えてみよう」 | 推測による修正。Phase 1に戻る |
| 「簡単な修正だからプロセスは不要」 | 簡単なバグにも根本原因がある |
| 「あとで調査する。今はこれで」 | 対症療法。根本原因を先に特定する |
| 「複数の修正をまとめて試す」 | 何が効いたか分からなくなる。1つずつ |
| 「もう1回だけ修正を試す」（2回失敗後） | 3回失敗 = アーキテクチャの問題 |
| 「テストなしで手動確認でいい」 | 自動テストなしの修正は信頼できない |

## Common Mistakes

| ミス | 対策 |
|------|------|
| エラーメッセージを読み飛ばす | 完全に読む。行番号・スタックトレースを確認 |
| 複数の変更を同時に行う | 1つずつ変更し、原因特定を可能にする |
| 修正前にテストを書かない | 回帰テストがないと再発を検出できない |
| 仮説なしにコードを変更 | 「なぜこれが原因か」を言語化してから変更 |
| git stash せずに変更 | 元に戻せるようにしてから変更する |

```markdown
- 症状: [...] / 再現手順: [...] / 影響範囲: [...]

## 調査過程
1. [調査1]: [結果]
2. [調査2]: [結果]

## 根本原因
- 直接原因: [...] / 根本原因: [...] / 原因箇所: [ファイル:行番号]

## 修正
- 方針: [根本修正/対症療法/回避策] / 変更: [...] / 追加テスト: [...]
```
