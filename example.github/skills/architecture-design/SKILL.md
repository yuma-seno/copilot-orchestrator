# アーキテクチャ設計スキル

要件定義に基づいて、実装可能なアーキテクチャ設計・コンポーネント設計を行います。

## 設計の手順

### ステップ1: 入力情報の確認

以下の情報を確認してから設計を開始してください：

1. **要件定義書**: `artifacts/requirements.md` またはユーザからの指示
2. **既存コードの構造**:
   - ディレクトリ構成とモジュール分割
   - 既存のアーキテクチャパターン（MVC、レイヤード、クリーンアーキテクチャ等）
   - 命名規則とコーディング規約
3. **技術スタック**: 使用中の言語・フレームワーク・ライブラリ
4. **既存の類似実装**: 参考にできる既存コード

### ステップ2: アーキテクチャ方針の決定

以下の観点からアーキテクチャ方針を決定してください：

#### 構造の決定

| 決定事項 | 検討ポイント |
|---------|------------|
| モジュール分割 | 既存の分割方針に合わせるか、新しいモジュールを追加するか |
| レイヤー構成 | プレゼンテーション / ビジネスロジック / データアクセスの分離 |
| 依存の方向 | 依存は内側（ビジネスロジック）に向かっているか |
| 公開インターフェース | 外部に公開するAPI・関数・クラスは何か |

#### パターンの選択

既存のコードベースで使用されているパターンを優先的に採用してください：

| パターン | 適用場面 |
|---------|---------|
| MVC / MVVM | UIのあるアプリケーション |
| レイヤードアーキテクチャ | 責務の分離が必要な場合 |
| リポジトリパターン | データアクセスの抽象化 |
| ファクトリパターン | オブジェクト生成の柔軟性 |
| オブザーバパターン | イベント駆動の処理 |
| ストラテジパターン | アルゴリズムの切り替え |
| ミドルウェアパターン | リクエスト処理のパイプライン |

### ステップ3: コンポーネント設計

各コンポーネントについて以下を定義してください：

```markdown
### コンポーネント: [名前]

**責務**: [このコンポーネントが担う責務（単一責任原則に基づく）]

**インターフェース**:
- 入力: [受け取るデータの型と説明]
- 出力: [返すデータの型と説明]
- 副作用: [ファイル書き込み、DB操作、API呼び出し等]

**依存関係**:
- [依存するコンポーネント名]: [依存の理由]

**エラーケース**:
- [エラー条件1]: [ハンドリング方針]
- [エラー条件2]: [ハンドリング方針]
```

### ステップ4: データフロー設計

データがシステム内をどう流れるかを明示してください：

```
[入力元] → [コンポーネントA]（処理内容） → [コンポーネントB]（処理内容） → [出力先]
```

**正常系フロー**: 主要な処理パスのデータの流れ
**異常系フロー**: エラー時のデータの流れ（エラーの伝播、ロールバック等）

### ステップ5: インターフェース定義

外部に公開するインターフェース（API、関数、クラス）を定義してください：

```markdown
### API / 関数 インターフェース

| 名前 | 入力 | 出力 | 説明 |
|------|------|------|------|
| [関数名/エンドポイント] | [引数/リクエストの型] | [戻り値/レスポンスの型] | [処理概要] |
```

### ステップ6: 影響範囲の分析

変更が既存コードに与える影響を分析してください：

```markdown
### 影響範囲

#### 新規作成するファイル
| ファイルパス | 内容 |
|------------|------|
| [パス] | [概要] |

#### 変更するファイル
| ファイルパス | 変更内容 | 影響度 |
|------------|---------|--------|
| [パス] | [変更概要] | 高/中/低 |

#### 影響を受ける既存機能
| 機能 | 影響内容 | 対策 |
|------|---------|------|
| [機能名] | [影響の説明] | [対策] |
```

### ステップ7: 設計書の出力

`artifacts/design.md` に以下のテンプレートで設計書を出力してください：

```markdown
# 設計書

## 1. 概要
[設計の全体概要]

## 2. アーキテクチャ方針
[選択したアーキテクチャパターンとその理由]

## 3. コンポーネント設計
[各コンポーネントの詳細（ステップ3の内容）]

## 4. データフロー
[正常系・異常系のデータフロー（ステップ4の内容）]

## 5. インターフェース定義
[外部インターフェースの定義（ステップ5の内容）]

## 6. エラーハンドリング方針
[エラーの種類、ハンドリング方法、ユーザへの伝え方]

## 7. 影響範囲
[影響範囲の分析結果（ステップ6の内容）]

## 8. 設計判断の記録
| 判断事項 | 選択肢 | 選択結果 | 理由 |
|---------|--------|---------|------|
```

## 設計原則

- **既存パターンの尊重**: プロジェクトで既に使われているパターンを優先する
- **単一責任原則**: 各コンポーネントは1つの責務のみを持つ
- **依存の最小化**: コンポーネント間の依存を最小限に抑える
- **テスタビリティ**: テストしやすい設計にする（依存の注入、純粋関数の活用）
- **段階的な変更**: 大きな変更は小さなステップに分割する
