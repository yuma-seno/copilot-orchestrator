# タスク計画スキル

ユーザの指示を分析し、実行可能なタスクに分解して計画を立てます。

## タスク分解の手順

### ステップ1: 指示の構造化分析

ユーザの指示から以下の要素を抽出してください：

**抽出する要素：**

| 要素 | 質問 | 例 |
|------|------|-----|
| 目的 | 最終的に何を達成したいか？ | 「ユーザ認証機能を実装する」 |
| 成果物 | どのような成果物が必要か？ | 「認証API、ログイン画面、テストコード」 |
| 制約 | 守るべき制約は何か？ | 「既存のDB構造は変更しない」 |
| 依存 | 外部の依存はあるか？ | 「OAuth2プロバイダとの連携」 |
| 品質基準 | 何を基準に完成とするか？ | 「全テスト通過、セキュリティ監査クリア」 |

**不明な要素がある場合：**
- オーケストレーションの場合: `askQuestion` でユーザに確認
- 単独作業の場合: 合理的な仮定を置き、判断をログに記録

### ステップ2: タスクの分解

指示を3つの粒度レベルで分解してください：

#### レベル1: フェーズ（Phase）

開発プロセスの大きな段階です。`skills/orchestration/workflow-selection.md` で選定されたワークフローに準じます。

```
例: 調査 → 要件定義 → 設計 → 実装 → テスト → レビュー
```

#### レベル2: 機能（Feature）

フェーズ内の機能単位の作業です。

```
例: 実装フェーズの場合
  - 認証APIエンドポイント
  - トークン管理モジュール
  - ログインフォームコンポーネント
```

#### レベル3: タスク（Task）

機能内の具体的な作業項目です。サブエージェントに1回の移譲で依頼可能な粒度にします。

```
例: 認証APIエンドポイントの場合
  - POST /api/auth/login エンドポイント実装
  - POST /api/auth/register エンドポイント実装
  - トークン検証ミドルウェア実装
```

**適切な粒度の判断基準：**

| 基準 | 適切 | 粗すぎる | 細かすぎる |
|------|------|---------|-----------|
| 作業時間目安 | 1つのやりとりで完了 | 複数回のやりとりが必要 | 複数タスクをまとめた方が効率的 |
| 完了判定 | 明確に判定可能 | 部分的にしか判定できない | 単独では意味をなさない |
| 責任範囲 | 明確に区切れる | 他タスクと境界が曖昧 | 過度に分割されている |

### ステップ3: 依存関係の整理

タスク間の依存関係を以下の3種類に分類してください：

| 依存タイプ | 説明 | 記法 |
|-----------|------|------|
| **直列** | 前のタスクが完了しないと開始できない | `A → B` |
| **並列** | 同時に実行可能（依存なし） | `A \| B` |
| **条件付き** | 特定の条件が満たされた場合のみ実行 | `A →? B`（条件: ...） |

**依存関係マップの例：**

```
task-001: DB スキーマ設計
  ↓
task-002: API エンドポイント実装  |  task-003: フロントエンドコンポーネント作成
  ↓                                    ↓
task-004: API テスト              |  task-005: フロントエンドテスト
  ↓                                    ↓
task-006: 結合テスト
  ↓
task-007: コードレビュー
```

### ステップ4: 優先度の設定

各タスクに優先度を設定してください：

| 優先度 | 基準 | 例 |
|--------|------|----|
| **P0（必須）** | クリティカルパス上。これが遅延すると全体が遅延。 | コアロジックの実装 |
| **P1（重要）** | 重要だが、他のタスクをブロックしない | テスト作成、エラーハンドリング |
| **P2（通常）** | あれば品質が向上するが、なくても動作する | ログ出力、コメント追加 |
| **P3（低）** | 時間があれば対応 | コードの微調整、ドキュメント更新 |

### ステップ5: タスク計画書の作成

以下のテンプレートでタスク計画書を作成してください：

```markdown
# タスク計画書

## 目的
[最終的に達成したいこと]

## スコープ
- 対象: [対象のファイル/機能]
- 対象外: [スコープ外のもの]

## フェーズ1: [フェーズ名]

### [Feature名]
- [ ] task-001: [タスク内容] (優先度: P0, 依存: なし)
  - 完了条件: [具体的な完了条件]
  - 対象ファイル: [ファイルパス]
- [ ] task-002: [タスク内容] (優先度: P0, 依存: task-001)
  - 完了条件: [具体的な完了条件]
  - 対象ファイル: [ファイルパス]

## フェーズ2: [フェーズ名]

### [Feature名]
- [ ] task-003: [タスク内容] (優先度: P1, 依存: フェーズ1完了)
  - 完了条件: [具体的な完了条件]

## 全体の完了条件
1. [プロジェクト全体の完了条件1]
2. [プロジェクト全体の完了条件2]

## リスクと対策
- [リスク1]: [対策]
- [リスク2]: [対策]
```

## 注意事項

- 不確実な部分は「調査タスク」として分離し、調査結果に応じて後続タスクを調整する
- タスク計画は固定的なものではなく、進捗に応じて柔軟に調整する
- 各タスクの完了条件は客観的に判定可能なものにする
