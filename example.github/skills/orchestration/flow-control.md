# フロー制御スキル

オーケストレーターが開発フローを制御するためのスキルです。

## 概要

タスクの進行状況に応じて、適切な次のアクションを決定します。

## 開発フローパターン

### 標準フロー

```
要件定義 → 設計 → 実装 → テスト → レビュー → 完了
```

### 反復フロー

レビューやテストの結果に基づき、前のフェーズに戻ります：

```
要件定義 → 設計 → 実装 → テスト → [問題あり] → 実装に戻る
                              → [問題なし] → レビュー → 完了
```

### 調査優先フロー

不確実性が高い場合に使用します：

```
調査 → 要件定義 → プロトタイプ → 検証 → 本実装 → テスト → レビュー → 完了
```

## フロー判定基準

### 次のフェーズに進む条件

| 現在のフェーズ | 進行条件 |
|---------------|----------|
| 要件定義 | 要件が明確で、ユーザの確認が取れている |
| 設計 | 設計方針が決まり、実装可能な状態 |
| 実装 | コードが完成し、基本動作が確認できる |
| テスト | 全テストが通過している |
| レビュー | 指摘事項がすべて解消されている |

### フェーズを戻す条件

- テストで重大な不具合が発見された → 実装フェーズに戻る
- レビューで設計上の問題が指摘された → 設計フェーズに戻る
- 要件の変更が発生した → 要件定義フェーズに戻る

## サブエージェントへの移譲判断

### 移譲する場合

- ファイルの内容確認が必要なとき
- 技術的な判断が必要なとき
- コードの実装・修正が必要なとき
- テストの実行・結果分析が必要なとき

### 自身で行う場合

- ユーザへの質問・確認（askQuestion）
- タスクリストの管理（todo）
- サブエージェントの呼び出し（agent）

## エラーハンドリング

### サブエージェントが失敗した場合

1. 失敗のサマリを確認
2. リトライ可能か判断
3. リトライする場合：追加の指示やコンテキストを添えて再移譲
4. リトライ不可の場合：ユーザに状況を報告し判断を仰ぐ

### 予期しない状況が発生した場合

1. セッションマネージャーに状況を記録
2. ユーザに状況を報告（askQuestion）
3. ユーザの指示に基づいてフローを調整
