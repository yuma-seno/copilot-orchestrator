# フロー制御スキル

タスク実行中のフロー制御（次のアクション判断、差し戻し、エラーハンドリング）を行います。

## フェーズ遷移の判断

### 次のフェーズに進む条件

サブエージェントの完了サマリを受け取った後、以下のチェックを行います：

```
サマリ受信
  ↓
結果の確認
  ├─ 結果: "成功"
  │    ↓
  │  品質ゲートチェック（skills/orchestration/quality-gate.md 参照）
  │    ├─ 全項目クリア → 次のフェーズへ進む
  │    ├─ 軽微な問題あり → サブエージェントに修正を指示し再チェック
  │    └─ 重大な問題あり → 前のフェーズに差し戻し
  │
  ├─ 結果: "失敗"
  │    ↓
  │  失敗原因の分析（後述「エラーハンドリング」参照）
  │
  └─ 結果: "要確認"
       ↓
     ユーザに確認（askQuestion）し、回答に基づいて判断
```

### フェーズ遷移マトリクス

| 現在のフェーズ | 成功時の次フェーズ | 差し戻し先 | 差し戻し条件 |
|---------------|-------------------|-----------|-------------|
| 調査 | 要件定義 | - | - |
| 要件定義 | 設計 | 調査 | 要件を満たす技術が不明 |
| 設計 | 実装 | 要件定義 | 要件の矛盾が発見された |
| 実装 | テスト | 設計 | 設計上の問題が発見された |
| テスト | レビュー | 実装 | テスト失敗、バグ発見 |
| レビュー | 完了 | 実装 or 設計 | 品質問題、設計問題 |

## 差し戻しの判断基準

### 差し戻す場合

以下のいずれかに該当する場合、前のフェーズに差し戻します：

1. **要件の矛盾・不整合**: 実装中に要件間の矛盾が発見された
   → 要件定義フェーズに戻る
2. **設計上の重大な問題**: 実装不可能な設計、パフォーマンス上の致命的問題
   → 設計フェーズに戻る
3. **テスト失敗**: 実装が要件を満たしていない
   → 実装フェーズに戻る（軽微な場合はそのフェーズ内で修正指示）
4. **レビュー指摘**: セキュリティ脆弱性、アーキテクチャ違反
   → 指摘の深刻度に応じて設計 or 実装フェーズに戻る

### 差し戻さない場合

以下の場合は、現在のフェーズ内で修正を指示します：

1. **軽微なコード品質問題**: 命名の改善、コメントの追加
2. **テストの不足**: 追加テストの作成
3. **ドキュメントの不備**: ドキュメントの更新
4. **フォーマットの問題**: コーディングスタイルの修正

## エラーハンドリング

### サブエージェントの失敗時

```
失敗サマリを受信
  ↓
失敗の種類を判別
  ├─ 一時的なエラー（ネットワーク、ファイルロック等）
  │    → 同じ指示でリトライ（最大2回）
  │
  ├─ 指示の不足・不明確
  │    → 追加のコンテキストを添えてリトライ
  │    → 必要に応じてユーザに確認
  │
  ├─ 技術的な困難（実現不可能、互換性問題等）
  │    → ユーザに状況を報告（askQuestion）
  │    → 代替案があればサブエージェントに調査させる
  │
  └─ スコープ外の問題
       → ユーザに報告
       → スコープの再検討が必要か判断
```

### リトライのルール

- **最大リトライ回数**: 2回（合計3回の試行）
- **リトライ時の追加情報**:
  - 前回の失敗ログのパスを伝える
  - 失敗原因に対する追加のヒントや制約を伝える
  - 必要に応じてスコープや完了条件を緩和する
- **3回失敗した場合**: ユーザに報告し、手動介入を依頼

### 予期しない状況の対処

1. `@sub-agent` に状況をセッションの `history` に記録させる
2. ユーザに状況を報告（`askQuestion`）
3. ユーザの指示に基づいてフローを調整
4. 必要に応じてワークフローを再選定（`skills/orchestration/workflow-selection.md` 参照）

## フローの動的変更

### 変更が必要な状況

- 作業中に要件の追加・変更が発生した
- 想定よりも問題が複雑であることが判明した
- ユーザからフローの変更を要求された

### 変更手順

1. 現在のタスク進捗を確認（`todo` の状態）
2. `@sub-agent` にセッション `history` に変更理由を記録させる
3. 新しいワークフローを選定（`skills/orchestration/workflow-selection.md` 参照）
4. `todo` を更新して新しいタスクリストを反映
5. セッションルールの質問粒度が `high` または `medium` の場合、ユーザにフロー変更を通知

## セッションルールに応じた制御

| セッションルール | フロー制御への影響 |
|----------------|-------------------|
| 質問粒度: high | フェーズ遷移ごとにユーザ確認。差し戻し前にもユーザ確認。 |
| 質問粒度: medium | 差し戻し時と予期しない状況でのみユーザ確認。 |
| 質問粒度: low | 重大な問題のみユーザに報告。差し戻しは自動判断。 |
| レビュー方針: auto | レビューフェーズはサブエージェントに自動実行させる |
| レビュー方針: manual | レビューフェーズでユーザに確認を求める |
| レビュー方針: skip | レビューフェーズをスキップして完了 |
