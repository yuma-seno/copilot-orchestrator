# copilot-orchestrator

VSCode GitHub Copilot向けのエージェントオーケストレーションシステム

## 概要

このリポジトリは、VSCode の GitHub Copilot で利用するためのエージェントオーケストレーションシステムのサンプル実装です。2つのエージェントと詳細なスキルセットを組み合わせ、開発タスクを効率的に遂行します。

## アーキテクチャ

```
ユーザ
  ↕ (askQuestion)
オーケストレーター [tools: askQuestion, todo | agents: sub-agent]
  ↕ (@sub-agent)
サブエージェント [tools: codebase, terminal, editFiles, fetchWebpage, useMcp]
  ↕ (skills参照)
ファイルシステム / .copilot/orchestrator/[session-id]/
```

### エージェント構成

| エージェント | 役割 | ツール | 呼び出し可能エージェント |
|-------------|------|--------|----------------------|
| **オーケストレーター** | セッション管理、ワークフロー選定、タスク移譲、品質ゲート判定 | askQuestion, todo | sub-agent |
| **サブエージェント** | 要件定義〜レビューまでの全実作業の実行 | codebase, terminal, editFiles, fetchWebpage, useMcp | なし |

### オーケストレーションフロー

```
1. セッション初期化（オーケストレーターが@sub-agentに指示）
2. セッションルール確認（ユーザにaskQuestion）
3. ワークフロー選定（指示内容に応じてスキル参照）
4. タスク計画（todoでタスクリスト作成）
5. 作業実行（@sub-agentに移譲→サマリ受け取り）
6. 品質ゲート判定（スキル参照→通過/差し戻し）
7. セッション完了（@sub-agentに完了処理を指示）
```

## セッションディレクトリ

オーケストレーションの実行時に、以下のディレクトリ構造が自動で作成されます：

```
.copilot/orchestrator/[yyyymmddhhmmssfff]/
├── session.json          # セッション情報（ルール、タスク、履歴）
├── summary.md            # セッション完了時の最終サマリ
├── logs/                 # タスク実行ログ
│   └── decisions/        # 判断記録
└── artifacts/            # 成果物（要件定義書、設計書、レビューレポート等）
```

## 使い方

### 1. セットアップ

`example.github/` フォルダの内容をプロジェクトの `.github/` フォルダにコピーします：

```bash
cp -r example.github/agents/ .github/agents/
cp -r example.github/skills/ .github/skills/
```

### 2. オーケストレーターの起動

VSCode の GitHub Copilot Chat で、オーケストレーターエージェントを呼び出します：

```
@orchestrator [開発タスクの指示]
```

### 3. セッションルールの確認

オーケストレーターが以下のルールを確認します：

- **質問粒度**: 高（都度確認）/ 中（要所で確認）/ 低（最小限の確認）
- **レビュー方針**: 自動 / 手動 / スキップ
- **スコープ**: 作業対象の範囲

### 4. 作業の実行

オーケストレーターがワークフローを選定し、タスクを計画して、サブエージェントに移譲します。
進捗はセッションディレクトリ内のログで確認できます。

### スキルの単独利用

各スキルはオーケストレーションシステム以外でも汎用的に利用できます。
例えば、サブエージェントを直接呼び出して特定のスキルを参照させることも可能です：

```
@sub-agent skills/development/code-review.md を参照して、このファイルをレビューしてください。
```

## 設計思想

### オーケストレーターの役割分離

オーケストレーターは「何をすべきか」を判断し、「どうやるか」はサブエージェントに委ねます。オーケストレーター自身はファイルの読み書きや技術的な判断を一切行いません。これにより：

- オーケストレーターのコンテキストウィンドウを節約
- サブエージェントが専門的な作業に集中可能
- セッションディレクトリを介した情報共有で、エージェント間の連携を実現

### セッションディレクトリによるコンテキスト管理

サブエージェントの詳細な出力はセッションディレクトリ内にファイルとして記録され、オーケストレーターには最小限のサマリのみが返されます。これにより：

- エージェント間の情報ロスを防止
- 作業の再現性と追跡可能性を確保
- 複数のサブエージェント呼び出し間でのコンテキスト引き継ぎを実現

### スキルによる作業手順の標準化

具体的な作業手順はスキルファイルに詳細に定義されており、エージェントはこれを参照して作業を進めます。各スキルには YAML フロントマターでメタデータ（名前、説明文、利用条件）が定義されており：

- 作業品質の均一化
- 手順の改善が容易
- エージェントへの指示を最小限に抑制
- オーケストレーションシステム以外でも汎用的に利用可能
