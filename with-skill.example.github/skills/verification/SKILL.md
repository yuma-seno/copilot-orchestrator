# Verification

タスクの完了を報告する前に、実際に検証を行う。

<HARD-GATE>
以下の全てを実行するまで「成功」「完了」「修正済み」と報告してはならない：
1. 検証コマンドを実際に実行した
2. コマンドの出力を確認した
3. 出力が期待結果と一致した
</HARD-GATE>

## 検証チェックリスト

タスクの種類に応じて、該当する検証を全て実行する：

### コード変更がある場合

- [ ] **ビルド** — ビルドコマンドを実行し、エラーがないことを確認
- [ ] **リンター** — リントコマンドを実行し、新規警告がないことを確認
- [ ] **テスト** — テストスイートを実行し、全テストが通過することを確認
- [ ] **動作確認** — 変更した機能が期待通りに動作することを確認

### バグ修正の場合

- [ ] 修正前に失敗していたテスト/操作が成功することを確認
- [ ] 修正が他の機能を壊していないことを確認（回帰テスト）

### 受け入れ基準がある場合

- [ ] 各受け入れ基準に対して、満たしていることを実際に確認

## 合理化の防止

以下の思考が浮かんだら、検証が不十分である：

| 思考 | 現実 |
|------|------|
| 「コードを見れば動くのは明らか」 | 明らかなコードも壊れる。実行して確認する |
| 「軽微な変更なのでテスト不要」 | 軽微な変更も他を壊す。実行して確認する |
| 「前回と同じパターンだから大丈夫」 | 前回と同じパターンでも壊れる。実行して確認する |
| 「手動で確認した」 | 手動確認は記録に残らない。コマンド実行で証拠を残す |
| 「テストを書いたから動くはず」 | テストを実行して通ることを確認する |

## 報告フォーマット

検証結果をサマリに含める：

```markdown
## 検証結果
- ビルド: [PASS/FAIL] — `[実行したコマンド]`
- テスト: [PASS/FAIL] — `[実行したコマンド]` ([通過数]/[全数])
- リント: [PASS/FAIL] — `[実行したコマンド]`
- 動作確認: [PASS/FAIL] — [確認内容と結果]
```
